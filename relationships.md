# Habibi — file relationships

A complete, file-by-file map of the project. For every file you'll see:

- **What it is** — role / purpose.
- **Imports** — what it depends on (the files it pulls in).
- **Imported by** — what depends on it.
- **Notes** — anything non-obvious about how it connects to the rest.

If a file is *not* listed in any "imported by" line, it's a **leaf** (entry point, generated artifact, or platform glue) — see §8 for those.

## 0. Reading guide

The repo splits into five categories:

| Category                   | Where                                         | Touched by you? |

| 1. Flutter app source      | `src/habibi/lib/`                             | yes — daily     |

## 2. Project root inside the Flutter project (`src/habibi/`)

These files describe the *project*, not its code.

### `src/habibi/pubspec.yaml`
- **What it is:** Flutter / Dart's package manifest. Lists app metadata, the SDK constraint, and every package the app pulls in.
- **Imports:** —
- **Imported by:** the `flutter` and `dart` CLIs use it on every command. Everything that says `import 'package:flutter/material.dart'` or `import 'package:hive/hive.dart'` only resolves because the package name is declared here.
- **Key contents:**
  - `flutter` (the framework)
  - `hive: ^2.2.3` and `hive_flutter: ^1.1.0` — local DB
  - `provider: ^6.1.2` — state management
  - `intl: ^0.20.2` — date formatting

### `src/habibi/pubspec.lock`
- **What it is:** the exact resolved versions of every transitive dependency. Auto-generated by `flutter pub get`.
- **Imported by:** the build tool, to make builds reproducible.
- **Notes:** never edit by hand.

### `src/habibi/analysis_options.yaml`
- **What it is:** lint configuration for the Dart analyzer.
- **Imported by:** the `dart analyze` and IDE checkers.
- **Notes:** controls warnings/errors during development. Does not affect runtime.

- **What it is:** Flutter's own bookkeeping — records which Flutter version + which platform plugins are configured.
- **Imported by:** `flutter create` / `flutter upgrade` consult it.
- **Notes:** auto-managed; usually checked in but not edited by hand.

### `src/habibi/.gitignore`
- **What it is:** tells git which paths to ignore (`build/`, `.dart_tool/`, etc.).

### `src/habibi/README.md`
- **What it is:** the auto-generated Flutter scaffold README ("A new Flutter project."). Mostly placeholder.

### `src/habibi/habibi.iml`, `src/habibi/.idea/...`
- **What it is:** IntelliJ / Android Studio project metadata (Dart SDK pointer, run configurations, workspace state).
- **Imported by:** the IDE only.
- **Notes:** has no effect on builds or runtime; safe to delete locally.

If you read just one diagram, this is it. **`main.dart` is the root**, everything else fans out from there. Models and utilities are at the bottom of the dependency chain — they import nothing project-internal except, in two cases, each other.

### `src/habibi/lib/main.dart`
- **What it is:** app entry point. Initializes Hive, registers the adapter, opens the box, runs the app.
- **Imports:**
  - `package:flutter/material.dart` — UI framework
  - `package:hive_flutter/hive_flutter.dart` — Hive's Flutter integration
  - `package:provider/provider.dart` — wraps the tree in `ChangeNotifierProvider`
  - `models/habit.dart` — needed to register `HabitAdapter`
  - `providers/habit_provider.dart` — wraps in `ChangeNotifierProvider<HabitProvider>`
  - `screens/home_screen.dart` — `home: const HomeScreen()`
- **Imported by:** nothing — this is the leaf at the top of the tree (Flutter's runtime calls `main()`).

### `src/habibi/lib/models/habit.dart`
- **What it is:** the `Habit` data class + its hand-written `HabitAdapter` for Hive serialization.
- **Imports:**
  - `package:hive/hive.dart` — for the `TypeAdapter` base class and `BinaryReader/BinaryWriter`.
- **Imported by:**
  - `main.dart` (registers adapter)
  - `providers/habit_provider.dart` (creates/edits/saves Habits)
  - `widgets/habit_card.dart` (receives a `Habit` to render)
- **Notes:** at the bottom of the layering — it depends on nothing app-internal. If you change the field list, both `read` *and* `write` of `HabitAdapter` must change in lockstep, or stored data won't deserialize.

### `src/habibi/lib/providers/habit_provider.dart`
- **What it is:** the only mutable layer. Wraps the Hive box, exposes `create`, `update`, `delete`, `toggleDay`, etc., and calls `notifyListeners()` so subscribed widgets rebuild.
- **Imports:**
  - `package:flutter/foundation.dart` — for `ChangeNotifier`
  - `package:hive/hive.dart` — for `Box<Habit>`
  - `package:uuid/uuid.dart` — fresh ID for each new habit
  - `models/habit.dart`
- **Imported by:**
  - `main.dart` (constructs it inside `ChangeNotifierProvider`)
  - `screens/home_screen.dart` (reads list)
  - `screens/habit_detail_screen.dart` (reads habit + toggles dates)
  - `screens/edit_habit_screen.dart` (creates / updates / deletes)
  - `widgets/habit_card.dart` (toggles today)
- **Notes:** this file is the choke point — every screen that wants to write goes through it. If you ever need to debug "why didn't the UI update?", set a breakpoint in `notifyListeners()` here.

### `src/habibi/lib/utils/date_key.dart`
- **What it is:** date↔string and date↔integer conversions. Pure functions, no state.
- **Imports:**
  - `package:intl/intl.dart` — for `DateFormat('yyyy-MM-dd')`
- **Imported by:**
  - `utils/streak.dart`
  - `widgets/habit_card.dart`
  - `widgets/dot_grid.dart`
  - `widgets/month_calendar.dart`
  - `screens/habit_detail_screen.dart`
- **Notes:** the most reused utility in the project. Anything that talks about "today", "this date", or "key for this date" goes through here.

### `src/habibi/lib/utils/streak.dart`
- **What it is:** `currentStreak` and `longestStreak` — two pure functions over a `Set<String>` of date keys.
- **Imports:**
  - `utils/date_key.dart`
- **Imported by:**
  - `screens/habit_detail_screen.dart` (the only consumer, used for the three stat chips)
- **Notes:** pure logic — easy to test in isolation if you ever decide to.

### `src/habibi/lib/utils/palette.dart`
- **What it is:** the project's design constants. 14 `Color`s and 30 `IconData`s.
- **Imports:**
  - `package:flutter/material.dart` (for `Color` and `IconData`)
- **Imported by:**
  - `screens/edit_habit_screen.dart` (passes lists into the pickers + reads icon code points)
  - `widgets/color_picker.dart`
  - `widgets/icon_picker.dart`
- **Notes:** the source of truth for colors and icons. If you want to add a new accent color or icon, this is the only file you edit.

## 4. The DSAP benchmark — `src/habibi/bench/`

Standalone Dart program. Has zero connection to `lib/` — they share no code. Lives in the same project only because it reuses Dart, the SDK, and (for tests) `flutter_test`.

### Import graph

```
                       ┌──────────────────────┐
                       │   bench_main.dart    │  (entry point — `dart run`)
                       └───────────┬──────────┘
                                   │ imports
        ┌──────────────────────────┼──────────────────────────┐
        ▼               ▼          ▼          ▼               ▼
 check_in_index   hash_set_index  sorted_array  bitmap_index  synthetic_data
        ▲               │             │            │
        │               └─── imports ─┴────────────┘
        └──────── imports ────────────┘
```

`check_in_index.dart` is the abstract interface. The three implementations all `implements CheckInIndex`. `bench_main.dart` is the only thing that imports all of them at once. `synthetic_data.dart` doesn't import any of the index files — it produces raw integers and lets the runner feed them into each implementation.

### `src/habibi/bench/check_in_index.dart`
- **What it is:** abstract class defining the shared API: `contains`, `add`, `remove`, `size`, `approxBytes()`, `label`.
- **Imports:** —
- **Imported by:**
  - `bench/hash_set_index.dart`
  - `bench/sorted_array_index.dart`
  - `bench/bitmap_index.dart`
  - `bench/bench_main.dart`
  - `test/algorithms_test.dart`
- **Notes:** this is the contract. All three implementations are interchangeable as long as they satisfy it.

### `src/habibi/bench/hash_set_index.dart`
- **What it is:** `Set<int>` wrapper. O(1) average. ~24 B/entry overhead.
- **Imports:** `bench/check_in_index.dart`
- **Imported by:** `bench/bench_main.dart`, `test/algorithms_test.dart`

### `src/habibi/bench/sorted_array_index.dart`
- **What it is:** `List<int>` kept sorted. `contains` = binary search (O(log n)). `add` = binary search + `List.insert` (O(n) shift).
- **Imports:** `bench/check_in_index.dart`
- **Imported by:** `bench/bench_main.dart`, `test/algorithms_test.dart`

### `src/habibi/bench/bitmap_index.dart`
- **What it is:** packed bit array (`Uint8List`). 1 bit per representable day. O(1) everything within the window; out-of-range writes silently dropped.
- **Imports:**
  - `dart:typed_data` (for `Uint8List`)
  - `bench/check_in_index.dart`
- **Imported by:** `bench/bench_main.dart`, `test/algorithms_test.dart`

### `src/habibi/bench/synthetic_data.dart`
- **What it is:** `buildWorkload(n, reps, seed)` produces reproducible random check-ins, queries, and add probes.
- **Imports:** `dart:math` (for `Random(seed)`)
- **Imported by:** `bench/bench_main.dart`
- **Notes:** the "random insert positions" detail in the add probes is what exposes SortedArray's O(n) cost. If probes were appended at the end, all three implementations would look equally fast on `add`.

### `src/habibi/bench/bench_main.dart`
- **What it is:** the runner. Loops over N ∈ {100, 1K, 10K, 100K} and the three implementations, times build / contains / add, prints a Markdown table, writes `bench/results.md` and `bench/results.csv`.
- **Imports:**
  - `dart:io` (for file output)
  - `bench/check_in_index.dart`
  - `bench/hash_set_index.dart`
  - `bench/sorted_array_index.dart`
  - `bench/bitmap_index.dart`
  - `bench/synthetic_data.dart`
- **Imported by:** —
- **Notes:** run with `dart run bench/bench_main.dart` from `src/habibi/`.

### `src/habibi/bench/results.md` / `bench/results.csv`
- **What it is:** the most recent benchmark output. Regenerated whenever you re-run `bench_main.dart`.
- **Imported by:** —

### `src/habibi/bench/README.md`
- **What it is:** the bench folder's own readme — a short note explaining what the benchmark does and how to run it.
- **Imported by:** —

## 6. The bridge between layers — how the Dart files actually talk at runtime

Imports tell you *what* one file knows about another. At runtime the connections look like this:

| Connection                                      | Mechanism                                                 |
|-------------------------------------------------|-----------------------------------------------------------|
| `main.dart` → `HabitProvider`                   | constructed inside `ChangeNotifierProvider` and lives at the top of the widget tree |
| Any widget → `HabitProvider`                    | via `Provider.of` / `context.read` / `context.watch` / `Consumer`. None of these touch the import graph — they walk the runtime widget tree. That's why `widgets/habit_card.dart` can fetch the provider without `main.dart` directly importing the card. |
| `HabitProvider` → `Hive.box`                    | the box is opened in `main.dart` and passed into the provider's constructor; the provider holds a reference. |
| `HabitProvider` → `Habit` on disk               | `_box.put(id, habit)` triggers `HabitAdapter.write`. `_box.get(id)` triggers `HabitAdapter.read`. |
| Screen A → Screen B                             | `Navigator.of(context).push(MaterialPageRoute(builder: (_) => B()))`. The import is needed because the builder constructs `B`. |
| Widget → repaint                                | `HabitProvider.notifyListeners()` → every `Consumer<HabitProvider>` and `context.watch` subscriber rebuilds. |

So the **import graph is mostly one-way** (UI imports state imports model), but the **runtime dependencies form a cycle**: state mutates → UI rebuilds → UI calls state. That's the Provider pattern.

## 7. Platform scaffolding — `android/`, `ios/`, `web/`, `windows/`

These folders are what turn your Dart code into actual installable apps for each OS. Flutter generated them when you ran `flutter create`. You almost never edit them.

| Folder       | Purpose                                                       | Built when                        |
|--------------|---------------------------------------------------------------|------------------------------------|
| `android/`   | Gradle project that wraps your Flutter engine for Android. APK is produced here. | `flutter run` / `flutter build apk` |
| `ios/`       | Xcode project for iPhone/iPad. **Requires macOS to build.**   | `flutter build ios` (Mac only)     |
| `web/`       | HTML shell + manifest + favicon for the web build.            | `flutter build web`                |
| `windows/`   | C++ Win32 wrapper for native Windows builds.                  | `flutter run -d windows` / `flutter build windows` |

### `src/habibi/android/`

The interesting files (you may eventually touch one of these):
- `android/app/build.gradle.kts` — Android build script. Where you'd change `applicationId`, `minSdk`, signing config, etc.
- `android/app/src/main/AndroidManifest.xml` — declares the app's permissions and entry activity.
- `android/app/src/main/kotlin/com/nvlznn/habibi/MainActivity.kt` — the Android entry point. Tiny — it just hands control to Flutter.
- `android/app/src/main/res/mipmap-*/ic_launcher.png` — launcher icons at every density.
- `android/app/src/main/res/values{,-night}/styles.xml` — splash screen styles.
- `android/gradle/wrapper/gradle-wrapper.{jar,properties}` — Gradle bootstrapper.
- `android/gradle.properties`, `android/gradlew`, `android/gradlew.bat` — Gradle entry points.
- `android/local.properties` — your local Android SDK path. **Do not commit if it contains absolute paths.**
- `android/.gitignore`, `android/app/build.gradle.kts`, `android/build.gradle.kts`, `android/settings.gradle.kts` — module config.

### `src/habibi/ios/`

You can't build any of this from your Windows machine — see the iOS build constraint memory.
- `ios/Runner.xcworkspace` and `ios/Runner.xcodeproj/...` — Xcode project files.
- `ios/Runner/AppDelegate.swift`, `SceneDelegate.swift` — iOS app entry points.
- `ios/Runner/Info.plist` — iOS app metadata.
- `ios/Runner/Assets.xcassets/AppIcon.appiconset/...` — every iOS icon size.
- `ios/Runner/Assets.xcassets/LaunchImage.imageset/...` — launch images.
- `ios/Runner/Base.lproj/{LaunchScreen,Main}.storyboard` — Interface Builder layouts for the launch screen.
- `ios/Runner/Runner-Bridging-Header.h` — Swift/Objective-C bridge header (mostly empty).
- `ios/Flutter/{AppFrameworkInfo.plist, Debug.xcconfig, Release.xcconfig}` — Flutter's iOS plumbing.
- `ios/RunnerTests/RunnerTests.swift` — iOS-side test placeholder (separate from your Dart tests).
- `ios/.gitignore` — iOS-specific ignores.

### `src/habibi/web/`
- `web/index.html` — the page that hosts the Flutter web canvas.
- `web/manifest.json` — PWA manifest.
- `web/favicon.png`, `web/icons/Icon-{192,512}.png`, `web/icons/Icon-maskable-{192,512}.png` — favicons / PWA icons.

### `src/habibi/windows/`
- `windows/CMakeLists.txt` and `windows/runner/CMakeLists.txt` — CMake build scripts.
- `windows/runner/main.cpp` — Windows entry point. Hands control to Flutter via `flutter_window.cpp`.
- `windows/runner/{flutter_window,utils,win32_window}.{cpp,h}` — Win32 plumbing that hosts the Flutter engine.
- `windows/runner/Runner.rc`, `windows/runner/runner.exe.manifest`, `windows/runner/resource.h`, `windows/runner/resources/app_icon.ico` — Windows resources (icon, manifest, version info).

A few things to notice:

- **`utils/date_key.dart` is the most-depended-on file in the project** (5 importers). If you ever changed its API, you'd ripple through almost the entire UI.
- **`utils/streak.dart` has exactly one consumer.** It could safely live inside `habit_detail_screen.dart` if you ever wanted — keeping it separate is a stylistic choice for testability.
- **`screens/settings_screen.dart` is the loneliest screen** — no internal imports in *or* out except being pushed by `home_screen`.
- **The bench folder is fully isolated.** No file under `lib/` imports anything in `bench/`, and vice versa. You can delete the entire `bench/` directory and the app still compiles and runs.
- **Tests are the only files that cross folders** — `test/algorithms_test.dart` reaches into `bench/` directly via relative imports.

## 10. If you change file X, what else might break?

| If you change…                                | Look at…                                                                                  |
|-----------------------------------------------|-------------------------------------------------------------------------------------------|
| Fields in `Habit`                             | `HabitAdapter.read` and `HabitAdapter.write` (must update *both*); `HabitProvider.create`/`update`; everywhere that constructs a `Habit` |
| `HabitAdapter` byte layout                    | All previously stored data on disk becomes unreadable. Bump `typeId` if you must.        |
| `HabitProvider` method signatures             | Every screen and `widgets/habit_card.dart` (they all call into the provider)             |
| `dateKey` / `epochDay` formats                | Stored data on disk + every utility/widget listed under `utils/date_key.dart`            |
| `habitColors` / `habitIcons` ordering         | Existing habits store `colorValue` / `iconCodePoint` directly, so reordering is fine; *removing* an entry orphans existing habits that pointed at it |
| `CheckInIndex` interface                      | All three implementations + `bench_main.dart` + `algorithms_test.dart`                   |
| `pubspec.yaml` version of `hive`              | Stored data may be incompatible — back up the box first                                   |
